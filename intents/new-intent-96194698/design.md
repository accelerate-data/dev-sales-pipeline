# Design: Monthly Product Aggregates

## Source Mapping

| Source Table | Staging Model | Mart Model | Status |
|-------------|---------------|------------|---------|
| `salescloud.opportunitylineitem` | `stg_salescloud__opportunitylineitem` | `fct_pipeline_monthly_product` | **New** |
| `salescloud.opportunity` | `stg_salescloud__opportunity` | `fct_pipeline_monthly_product` | **Existing** |

**Build sequence:**
1. Create `stg_salescloud__opportunitylineitem` (staging layer)
2. Create `fct_pipeline_monthly_product` (mart layer)

## Model Architecture

```
Sources (Bronze)
└── salescloud.opportunitylineitem
└── salescloud.opportunity

Staging (Silver) 
└── stg_salescloud__opportunitylineitem  [NEW]
└── stg_salescloud__opportunity          [EXISTING]

Marts (Gold)
└── fct_pipeline_monthly_product         [NEW]
    ├── joins: stg_salescloud__opportunitylineitem
    └── joins: stg_salescloud__opportunity
```

**Dependency flow:**
- `fct_pipeline_monthly_product` depends on `stg_salescloud__opportunitylineitem` and `stg_salescloud__opportunity`
- Both staging models are views (fast, no storage cost)
- Mart is a table (materialized for query performance)

## Materialization Strategy

| Model | Materialization | Rationale |
|-------|----------------|-----------|
| `stg_salescloud__opportunitylineitem` | `view` | 1:1 with source, minimal transformation, follows project convention for staging |
| `fct_pipeline_monthly_product` | `table` | Aggregated data, pre-calculated metrics, frequently queried for dashboards - table materialization improves query performance |

**Future consideration:** If line item data grows very large (millions of rows), consider `incremental` materialization for the staging model with `CreatedDate` as the incremental key.

## Validation Approach

### Row Count Validation
- Count of distinct (month, product) combinations in mart should be ≤ count of distinct products × number of months in data
- Total revenue in mart should equal sum of TotalPrice from line items (when filtered to non-null close dates)

### Metric Validation
- For any specific product-month: manually verify that summed revenue matches
- Opportunity count for a product-month should be ≤ line item count (one opp can have multiple line items)
- Average deal size should = total revenue / opportunity count (validated via dbt test)

### Data Quality Checks
- No negative quantities or prices
- No future dates in close_month
- All product IDs should exist in source
- UnitPrice * Quantity should approximately equal TotalPrice (accounting for discounts)

## Grain Specification

### stg_salescloud__opportunitylineitem
**Grain:** One row per opportunity line item  
**Primary Key:** `line_item_id` (Id from source)

### fct_pipeline_monthly_product
**Grain:** One row per calendar month + product combination  
**Primary Key:** `close_month || '_' || product_id` (composite key)  
**Example row:** "2026-03" + "PROD-001" = one row with aggregated metrics for Product PROD-001 in March 2026

## Join Plan

### Staging Model (stg_salescloud__opportunitylineitem)
No joins - direct 1:1 mapping from source table with column renaming and type casting.

### Mart Model (fct_pipeline_monthly_product)

```sql
line_items (stg_salescloud__opportunitylineitem)
  INNER JOIN opportunities (stg_salescloud__opportunity)
    ON line_items.opportunity_id = opportunities.opportunity_id
  WHERE opportunities.close_date IS NOT NULL
  GROUP BY 
    DATE_TRUNC('month', opportunities.close_date) as close_month,
    line_items.product_id,
    line_items.product_code,
    line_items.product_name
```

**Join type rationale:**
- **INNER JOIN** is appropriate because:
  - We need close_date from opportunities to determine the month
  - Line items without a corresponding opportunity are orphaned data (data quality issue)
  - Line items with NULL close_date cannot be assigned to a month
  
**Fan-out risk:** None - joining line items to opportunities is many-to-one (many line items per opportunity)

**Performance consideration:** The join is on `opportunity_id` which should be indexed in the source system. If performance issues arise, consider adding a filter on `close_date` to reduce the opportunity scan.

## Testing Plan

### Schema Tests (Generic Tests)

**stg_salescloud__opportunitylineitem:**
- `not_null` on `line_item_id`, `opportunity_id`
- `unique` on `line_item_id`
- `relationships` to `stg_salescloud__opportunity` on `opportunity_id`

**fct_pipeline_monthly_product:**
- `not_null` on `close_month`, `product_id`
- `unique` on composite key (`close_month` + `product_id`)
- `accepted_values` on `close_month` (format: YYYY-MM-DD for first of month)

### Data Quality Tests

Will be generated by data-test-writer sub-agent, expected tests include:
- Total revenue > 0 for all rows
- Quantity >= 0
- Opportunity count >= 1
- Average deal size = total revenue / opportunity count
- No future months (close_month <= current_date)

### Unit Tests

Will be generated by unit-test-writer sub-agent based on edge cases identified during build.

## Data Quality Rules

### Needing Attention (User Review Required)

| Model | Rule | Package | Rationale |
|-------|------|---------|-----------|
| `fct_pipeline_monthly_product` | Revenue consistency check | Custom | Verify that sum of monthly product revenue equals sum of all line items with non-null close dates. This catches any join issues or filtering problems. |
| `fct_pipeline_monthly_product` | Quantity-price consistency | Custom/dbt-expectations | For each row, verify that total_quantity * avg_unit_price is approximately equal to total_revenue (within discount tolerance). This validates our aggregation logic. |
| `fct_pipeline_monthly_product` | Orphaned products detection | Custom | Flag any product IDs that appear in line items but don't appear in any monthly aggregates (indicates missing close dates). |

### Default (Schema Integrity)

- `unique`, `not_null` on primary keys (line_item_id for staging, close_month+product_id for mart)
- `relationships` on foreign keys (opportunity_id, product_id)
- `accepted_values` on close_month format

## Unit Test Scenarios

| Model | Scenario | What It Tests |
|-------|----------|---------------|
| `fct_pipeline_monthly_product` | Multiple line items for same product in same month | Aggregation logic correctly sums quantities, revenues, and counts distinct opportunities |
| `fct_pipeline_monthly_product` | Product sold across multiple months | Each month gets its own row, no cross-month contamination |
| `fct_pipeline_monthly_product` | Line item with NULL close_date | Filtered out (doesn't appear in any monthly aggregate) |
| `fct_pipeline_monthly_product` | Multiple line items for same product in same opportunity | Opportunity count is 1, but quantity and revenue are summed from both line items |
| `fct_pipeline_monthly_product` | Line items with discounts | TotalPrice is used (already includes discount), avg_unit_price is calculated correctly |
| `fct_pipeline_monthly_product` | Closed Won vs Closed Lost | Both are included in aggregates (can be filtered downstream) |
